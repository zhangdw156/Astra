# 数据合成项目技术路线方案（改进版）

## 一、项目总体目标与设计理念

本项目旨在构建一套端到端的数据合成流水线，围绕具备工具调用能力的Skill，自动化产出高质量的多轮工具调用数据以及可运行的工具环境。项目的核心价值在于将传统的人工数据标注模式转变为自动化、可扩展的流水线作业，通过结构化的质量门控机制确保合成数据的可用性与安全性，同时通过引入数据库驱动的状态管理机制，解决有状态环境中的状态一致性、可复现性和可验证性难题。

整体设计遵循以下核心理念。首先是模块化解耦原则，各阶段通过JSON等标准化数据格式进行连接，使得每个阶段可以独立演进、灵活组合，这种设计借鉴了软件工程的最佳实践，使得流水线的各个组件可以独立测试和优化。其次是质量优先原则，在关键环节设置验证与评估节点，形成多层次的质量门控体系，确保问题在早期被发现和修复，降低后期修复成本。第三是状态驱动原则，引入数据库作为环境状态的后端，为每个环境提供结构化、可追踪、可验证的状态管理能力，这一设计直接借鉴了Agent World Model的核心思想。第四是可复现性保证，通过环境隔离与确定性状态初始化机制，确保每次合成任务的结果可复现、可追溯，这对于大规模数据合成和质量控制至关重要。

本方案在原有技术路线的基础上进行了系统性升级，主要改进点包括：引入四阶段渐进式环境生成流程替代单一步骤的代码生成；采用SQLite数据库作为状态后端实现环境状态的确定性和可验证性；引入POMDP形式化框架对环境进行数学建模；增加自验证与自动修复机制提升生成质量；引入双Agent质量评估体系实现自动化的环境验收；采用状态优先的任务生成策略确保任务可解性。在蓝图生成与代理模拟方面，借鉴 LoopTool 与 APIGen-MT 的思路，将静态的 query 列表改为两阶段蓝图生成（任务配置 + 交互轨迹生成）与多Agent协作，代理模拟阶段采用动态用户模拟与多轮交互，使合成数据更贴近真实人机对话。这些改进使得流水线在规模化能力和生成质量方面都有显著提升。

## 二、流水线核心架构

整个数据合成流水线由五个核心阶段串联而成，形成从原始Skill定义到最终可用数据的完整数据流。各阶段之间通过标准化的数据结构进行交互，上游阶段的输出作为下游阶段的输入，形成清晰的依赖关系。与原有方案相比，改进后的架构在环境生成阶段内部进行了更细粒度的分解，引入了场景规划、数据库合成、工具生成和自验证修复四个子阶段。

第一阶段是环境生成阶段，负责将Skill定义转换为可运行的MCP工具环境。该阶段以Skill目录和参考示例为输入，通过四阶段渐进式流程生成包含数据库Schema、初始状态数据、MCP服务器、工具实现、Mock服务和容器配置在内的完整环境代码。第二阶段是环境验证阶段，对生成的环境进行结构检查、依赖安装、运行时探针验证和功能测试，只有通过验证的环境才会进入下一阶段。第三阶段是蓝图生成阶段，采用两阶段框架（任务配置 + 交互轨迹生成）和多Agent协作（Planner、User、Assistant、Tool），在带验证的迭代中产出任务蓝图与交互生成配置，而非一次性生成静态query列表。第四阶段是代理模拟阶段，采用动态用户模拟机制：由用户Agent根据对话上下文与任务目标逐轮生成用户消息，与助手Agent多轮交互直至任务完成，并对轨迹进行状态与输出双重验证与筛选。第五阶段是轨迹评估阶段，对生成的轨迹进行质量评分和风险评估，过滤低质量数据并输出最终的合成数据集。

各阶段之间通过标准化的数据结构进行解耦。环境生成阶段输出包含数据库文件、工具代码和容器配置的完整环境包；蓝图生成阶段输出包含任务配置（用户意图、期望工具调用序列、期望状态与输出）和交互生成配置（用户Agent配置、最大轮次、结束条件）的结构化JSON；代理模拟阶段输出由动态用户模拟产生的、含完整对话历史与工具执行记录的轨迹数据；轨迹评估阶段输出包含质量评分、风险标签和改进建议的评估报告。这种标准化的接口设计使得各阶段可以独立演进，当某一阶段的生成质量提升时，整个流水线的输出质量都会相应提升。

## 三、环境生成

### 3.1 技术架构与核心思路

环境生成阶段采用四阶段渐进式合成架构，这一设计借鉴了Agent World Model和EnvScaler的核心思想，同时针对已有的Skill到Environment转换框架做了适配和增强。四阶段分别为场景规划阶段、数据库与状态合成阶段、工具与接口合成阶段、自验证与修复阶段。每个阶段都有明确的输入输出和验证标准，通过这种结构化的方法可以将复杂的环境生成问题分解为可管理、可验证的子问题。

场景规划阶段接收Skill目录作为输入，输出包含环境类型分类、状态空间规划和工具需求清单的结构化文档。该阶段的核心任务是分析Skill的特征，确定环境类型，并规划状态空间。场景规划借鉴了EnvScaler的任务驱动环境发现思路，通过解析SKILL.md的内容来提取功能领域、涉及的外部依赖、命令和工具列表，然后基于这些信息对Skill进行分类。分类维度包括状态复杂度（无状态、轻量状态、复杂状态）、交互模式（纯API调用、文件操作、数据库操作）、领域类型（金融、社交、 productivity、开发工具等）。状态空间规划是该阶段的核心输出，它以结构化的方式描述环境需要管理的实体类型、实体属性和实体间关系。

数据库与状态合成阶段接收场景规划的输出作为输入，输出包含SQLite Schema定义文件和初始状态数据的数据库文件。该阶段的设计直接借鉴了Agent World Model的核心创新，即使用SQLite作为环境状态的后端。传统的基于文件系统或Mock服务的方式存在状态不一致的问题，而数据库驱动的状态管理可以提供可靠的状态转换。Schema设计由大语言模型根据状态空间规划自动生成，模型需要推断所需的实体类型、实体属性、实体间关系，并将其转换为符合SQLite规范的Schema定义。初始状态数据合成采用状态优先策略，先生成能够支撑任务执行的初始数据，再基于这些数据派生任务需求。对于需要高并发合成的环境，推荐进一步将状态拆分为**共享静态状态**与**运行态状态**两层：共享静态状态保存所有轨迹共用的只读种子数据，运行态状态则通过 `run_id` 作为逻辑分区键记录每条轨迹的会话元数据、工具调用日志、状态快照与中间输出。这样既保留数据库驱动的可验证性，又避免为每条只读轨迹复制整份数据库。

工具与接口合成阶段接收数据库Schema和初始状态作为输入，输出包含MCP服务器、工具实现和Mock服务的完整代码。该阶段在原有方案的基础上进行了增强，工具实现通过统一的状态访问层来操作数据库，而非直接访问外部服务或文件系统。状态访问层封装了SQLite数据库的读写接口，提供了read、write、delete和transaction等核心方法。这种设计有几个重要优势：状态变更可追踪和回滚、便于实现事务性操作、支持状态验证。对于批量数据合成场景，状态访问层还应提供 `run_id` 作用域管理能力，由上层框架在启动单条轨迹时创建 `run_id` 并注入 `RunContext`，随后由状态层自动将运行态读写绑定到当前 `run_id`。这样工具 schema 无需暴露 `run_id` 参数，工具实现也能保持简洁。Mock服务的设计也做了相应调整，Mock不再返回静态JSON，而是从同一个SQLite数据库中读取数据返回给工具，这样可以保证工具看到的数据与环境的实际状态一致。

在具体工程实现上，本项目在**环境生成阶段只关注一种输出形态：轻量工具环境（light/json-only）**。

- **轻量环境（light/json-only）**（流水线生成目标）：环境目录不强制要求 `database/`、`state.py`、`tools/`，而只要求 `mcp_server.py`、`pyproject.toml` 与 `tools.jsonl`。MCP 服务根据 `tools.jsonl` 注册通用工具 handler，通过 LLM 在进程内的 KV 中维护 JSON 会话状态（即「环境 state」），每次工具调用后产生新的完整 state，并在轨迹结束时写入 `final_state`。这种模式适合只读或弱状态场景，环境极简、易于批量生成和校验，同时遵循统一的 `run_id` / `final_state` 协议，便于在评估与分析阶段与其他环境形态对齐。

自验证与修复阶段是实现高质量生成的关键机制，借鉴了Agent World Model的自我纠正设计理念。该阶段执行多轮验证和修复循环，直到环境通过所有验证或达到最大迭代次数。验证内容包括结构验证、依赖验证、运行时验证和功能验证四个层面。自动修复机制在检测到错误后会自动分析错误类型，将错误描述和上下文反馈给大语言模型进行修复，修复后的代码会重新经过验证流程。

### 3.2 场景规划详细设计

场景规划是改进方案新增的核心环节，它为后续的环境生成提供了方向指引和约束条件。一个完善的场景规划需要包含以下几个关键组成部分。

环境类型分类是场景规划的起点。通过分析SKILL.md的内容，系统会判断该Skill对应的环境属于哪种类型。环境类型的划分影响着后续生成策略的选择。对于无状态环境，主要涉及纯信息查询或计算功能，不需要持久化状态，可以简化数据库设计。对于轻量状态环境，涉及简单的用户会话或临时状态，可以使用内存数据库或简化的表结构。对于复杂状态环境，涉及多实体关联和复杂业务逻辑，需要完整的关系型数据库Schema设计。环境类型的分类还可以进一步细化为API调用型、文件操作型、数据库操作型、混合型等，以便选择最相似的参考示例。

状态空间规划是场景规划的核心输出。一个完整的state_space_description包含以下信息：实体类型清单列出环境中需要管理的所有实体，如用户、订单、产品、文件等；实体属性描述每个实体包含的属性及其数据类型和约束条件；实体间关系描述不同实体之间的关联方式，如一对多、多对多等；操作映射描述每个工具操作对应哪些实体的哪些状态变更。这种结构化的状态空间规划为后续的数据库Schema设计和工具实现提供了明确的约束。

工具需求分析从Skill描述中提取需要实现的工具列表，并为每个工具标注其状态依赖关系。工具可以分为几类：只读型工具仅查询状态，不产生状态变更；写入型工具产生状态变更；复合型工具先查询再写入。标注工具的状态依赖关系有助于后续的数据库Schema设计和验证函数生成。

### 3.3 数据库与状态合成详细设计

数据库与状态合成阶段将场景规划的输出转换为可执行的数据库实现。该阶段的设计直接借鉴了Agent World Model的核心创新，即使用编程方式（而非LLM模拟）来管理环境状态，从而实现状态转换的确定性和一致性。

Schema生成是该阶段的第一个关键步骤。系统调用大语言模型，根据状态空间规划中描述的实体类型、属性和关系生成SQLite的CREATE TABLE语句。生成的Schema需要满足几个约束：能够支持Skill中定义的所有工具操作、包含任务执行所需的所有数据字段、定义合理的约束条件（如外键、唯一性、非空等）。Schema生成还需要考虑查询性能，对于高频查询的字段应建立索引。

初始状态数据合成是保证任务可解性的关键步骤。这一设计借鉴了EnvScaler的“状态优先”理念，即先生成初始状态，再基于初始状态派生任务。初始状态的生成需要满足几个条件：包含任务执行所需的基础数据（如用户账户、示例文件等）、数据之间的关联关系一致、能够支撑工具的正常执行而不产生空引用错误。初始状态以SQL INSERT语句的形式输出，可以直接用于初始化数据库。为了实现可复现性，初始状态的生成使用确定性随机算法，相同的Skill和任务ID总是产生相同的初始数据。

数据库层的设计还需要考虑状态访问层的接口定义。状态访问层是连接工具实现和SQLite数据库的桥梁，它封装了所有状态相关的操作。核心接口包括：read方法接收实体类型和查询条件，返回匹配的数据库记录；write方法接收实体类型和要写入的数据，执行INSERT或UPDATE操作；delete方法接收实体类型和查询条件，执行DELETE操作；transaction方法提供事务支持，确保多个操作的原子性。

### 3.4 工具与接口合成详细设计

工具与接口合成阶段生成MCP服务器和工具实现代码。与原有方案的关键区别在于，工具实现通过状态访问层来操作数据库，而非直接访问外部服务或文件系统。

MCP服务器的实现保持与原有方案一致的结构，使用FastMCP框架，支持工具的动态发现和注册。服务器启动时扫描tools/目录下的所有Python文件，加载包含TOOL_SCHEMA和execute函数的模块，并将其注册为MCP工具。服务器通过环境变量MCP_TRANSPORT配置传输方式，支持stdio和SSE两种模式。

工具实现的关键改进在于execute函数的内部逻辑。原有的execute函数直接调用外部API或操作文件系统，改进后的execute函数通过状态访问层来完成任务。状态访问层将工具的操作意图转换为数据库的读写操作，并记录状态变更历史。这种设计有几个重要优势：所有状态变更都是可追踪的、便于实现状态回滚、支持事务性操作、便于编写验证函数。

Mock服务的设计也需要相应调整。当Skill涉及第三方服务依赖时，Mock服务不再返回静态JSON响应，而是从同一个SQLite数据库中读取数据。具体做法是：Mock服务作为数据库的只读客户端，根据请求参数查询数据库并返回结果。这样可以保证工具看到的数据与环境的实际状态一致，避免静态Mock数据与实际状态不一致的问题。

### 3.5 POMDP形式化框架

为了实现环境的数学化建模和可验证性，每个生成的环境都可以形式化为一个POMDP（部分可观测马尔可夫决策过程）。这种形式化借鉴了Agent World Model的理论框架，为环境的行为提供了严格的数学定义。

状态空间（S）由SQLite数据库的Schema定义，数据库中的每一行记录对应状态空间中的一个具体状态点。状态空间的设计需要覆盖工具操作可能触及的所有数据范围，包括实体数据、关系数据和元数据。

动作空间（A）是MCP工具调用的集合，每个工具调用对应动作空间中的一个可执行动作。动作空间的设计需要覆盖Skill中定义的所有命令和功能，每个工具都应该能够被代理调用。

观测空间（O）是工具执行后返回的结果，这些结果构成了代理感知环境变化的唯一途径。由于工具返回的是数据库查询结果或操作反馈，观测空间是确定性的，给定相同的状态和动作必然产生相同的观测。

转移函数（T）通过数据库的读写操作实现。工具的execute函数内部调用状态访问层的write或delete方法，这些方法修改数据库状态。转移函数是确定性的，这是数据库驱动方法相对于LLM模拟方法的核心优势。

奖励函数（R）由验证代码定义。EnvScaler和Agent World Model都采用了程序化验证函数而非LLM判断的方式。验证函数接收任务描述和执行前后的数据库状态，返回任务完成程度的评分。这种设计避免了LLM判断的不一致性和主观性。

### 3.6 自验证与修复机制

自验证与修复机制是实现高质量生成的关键，借鉴了Agent World Model的自我纠正设计。该机制通过多轮验证和修复循环，不断提升生成环境的质量，直到通过所有验证标准。

验证流程包含四个层面的检查。结构验证检查必要的文件和目录是否完整，包括mcp_server.py、tools/目录、pyproject.toml、Dockerfile、docker-compose.yaml、database schema文件和初始数据文件等。依赖验证执行uv sync命令，确认所有Python包能够正确安装。运行时验证启动MCP服务器并发送Initialize探针请求，确认服务器能够正常响应。更重要的是功能验证，执行测试用例检查工具是否能正常工作，工具执行前后的状态变更是否符合预期。

自动修复机制在检测到错误后会自动尝试修复。修复流程如下：首先分析错误类型，判断是结构问题、代码问题还是逻辑问题；然后将错误描述、错误位置和上下文信息组织成修复提示；接着调用大语言模型生成修复建议；最后将修复建议应用到代码中并重新验证。修复循环会持续进行，直到验证通过或达到最大迭代次数（通常设置为5次）。根据Agent World Model的实验数据，这种机制可以实现超过85%的首次成功率，平均修复迭代次数约为1.13次。

### 3.7 双Agent质量评估

除了自验证机制，还可以引入EnvScaler的双Agent设计进行更深入的质量评估。这种评估机制在环境生成完成后执行，不依赖于错误发生后的被动修复，而是主动检测环境的潜在问题。

测试Agent模拟代理的行为，执行以下任务：读取工具的描述和接口定义、根据任务描述生成工具调用请求、观察工具执行的结果和状态变化。测试Agent采用“黑盒”测试方法，不需要了解工具的内部实现，只根据工具的外部接口进行操作。这种测试方式能够验证工具的外部行为是否符合预期。

检查Agent负责评估测试结果，执行以下任务：检查工具执行的返回结果是否符合规范、检查状态变化是否符合预期、检查是否存在错误或异常、检查工具是否正确处理了边界情况。检查Agent可以访问工具的源代码和数据库状态，实现“白盒”检查。

双Agent评估会迭代进行多轮，每轮生成不同的测试用例，计算通过率作为质量指标。只有通过率达到预设阈值（通常设置为0.85）的环境才会被标记为可用。评估结果还会生成详细的报告，指出环境的优点和不足，为后续的优化提供方向。

## 四、环境验证

### 4.1 验证流程设计

环境验证是保证数据合成质量的关键环节，采用分层验证策略，在不同层次设置质量门。改进后的验证流程在原有三层验证（结构、依赖、运行时）的基础上，增加了功能验证和数据库状态验证两个层次。

结构验证层检查环境目录的完整性。首先验证必要的入口文件和配置文件是否存在，包括mcp_server.py、pyproject.toml、Dockerfile、docker-compose.yaml、数据库Schema文件、初始数据文件等。其次验证工具目录的结构是否规范，tools/目录下是否包含必要的__init__.py文件，每个工具文件是否定义了TOOL_SCHEMA和execute函数，工具名称是否与文件名一致。最后验证tools.jsonl的内容是否与实际工具文件一致。

依赖验证层在隔离环境中安装项目依赖。使用uv sync --no-install-project命令在干净环境中安装依赖，捕获任何安装失败的错误信息并记录。依赖验证还会检查Python版本兼容性，确保生成的代码在目标Python版本上能够正常运行。

运行时验证层启动MCP服务器并执行健康检查。首先安装项目自身的依赖，然后启动MCP服务器进程，最后发送MCP Initialize请求或等价的探针请求。如果服务器能够在合理时间内响应，则认为运行时验证通过。

功能验证层执行基础的功能测试。启动环境中的所有组件（数据库、MCP服务器、Mock服务），执行预设的测试用例，检查工具的基本功能是否正常。功能验证关注工具是否能够正确执行、返回结果格式是否正确、错误处理是否合理。

数据库状态验证层检查工具执行前后的状态变更是否符合预期。这一验证需要结合验证函数进行，验证函数定义了任务完成条件的检查逻辑。通过比较工具执行前后的数据库状态，验证函数可以判断操作是否产生了预期的效果。

### 4.2 验证结果结构

每个环境的验证结果采用结构化格式存储，包含以下字段：环境标识信息包括环境名称、Skill来源、生成时间戳和版本号；验证状态标记为通过、失败或部分通过；各层验证的详细结果包含具体的错误信息、错误位置和堆栈跟踪；验证耗时统计包括各层验证的耗时和总耗时；修复历史记录包含每次修复尝试的内容和结果。

批量环境的验证结果在此基础上增加汇总统计信息，包括整体通过率、各失败原因的出现次数和占比、通过LLM生成的修复建议等。修复建议可以根据验证失败的错误模式自动生成，指出最可能的修复方向和具体操作步骤。

### 4.3 可选的报告生成

对于批量验证场景，可以启用LLM报告生成能力。该功能将验证结果汇总后输入到大语言模型，生成自然语言形式的分析报告。报告内容包括验证整体概况（通过率、失败率、平均验证时间）、失败环境清单及原因分类（按错误类型聚类）、修复优先级建议（根据错误影响范围和修复难度排序）、以及针对常见问题的通用解决方案。

报告生成还可以包含趋势分析内容，比较不同批次或不同时间段的验证结果，发现潜在的系统性问题。例如，如果某个特定错误类型频繁出现，可能提示环境生成的提示词或参考示例存在问题，需要相应调整。

## 五、蓝图生成

### 5.1 核心改进思路与蓝图结构

原有方案在蓝图生成阶段一次性生成所有 user query，在代理模拟阶段按顺序执行这些预定义 query。这种静态方法存在明显局限：生成的 query 可能不够自然、无法根据代理的实际响应动态调整、缺乏真实的人机交互感。借鉴 LoopTool 的多 Agent 协作生成对话与 APIGen-MT 的两阶段框架与模拟人机交互，本方案将静态的 query 列表转变为**任务配置 + 动态交互生成**：蓝图只给出任务核心要素与交互生成配置，具体每轮用户消息由代理模拟阶段的用户 Agent 根据上下文动态生成，使合成数据更接近真实人机对话。

蓝图是连接环境定义和代理执行的关键中间产物，以 JSON 格式存储，包含任务配置与交互生成配置两部分，供下游代理模拟阶段使用。

**任务配置部分**（对应两阶段中的“任务配置阶段”产出）包含：task_id（任务唯一标识）；goals（有序用户目标）；possible_tool_calls（可能的工具调用，与 goals 一一对应，用于评估与验证）；expected_final_state（任务完成后的期望状态描述）。该部分不包含具体对话内容，仅作为交互生成的指导蓝图。

**交互生成配置部分**（指导“交互轨迹生成阶段”与代理模拟）包含：system_message（助手 Agent 的系统提示词，含角色、能力边界、响应风格）；user_agent_config（用户 Agent 的配置，如角色、性格、知识边界，用于动态生成用户消息）；max_turns（对话最大轮次）；end_condition（结束条件判断逻辑，如任务达成、用户主动结束）。可选的 initial_state_override 允许在默认初始状态基础上微调，以生成更多样化的任务场景。

蓝图的顶层元信息包括：blueprint_id、skill_name、skill_version、persona_id、created_at、environment_path，含义与原有方案一致。

### 5.2 两阶段蓝图生成框架

借鉴 APIGen-MT 的两阶段设计，蓝图生成分为**任务配置阶段**和**交互轨迹生成阶段**。

**任务配置阶段**负责产出任务蓝图的核心要素，包含高层用户意图、期望的工具调用序列和预期输出。本阶段不生成完整对话，只生成：用户目标的抽象描述、完成任务所需的关键动作序列、以及任务完成后的期望状态。这些信息作为后续交互生成的指导蓝图，而非直接执行的脚本。生成时仍采用状态优先策略：分析数据库初始状态，从初始状态派生可行任务，确保任务可解。

**交互轨迹生成阶段**负责将任务蓝图转化为完整的多轮对话轨迹。本阶段不再一次性生成所有 user query，而是为代理模拟阶段配备“用户 Agent”：在代理模拟时，由用户 Agent 根据任务意图和对话上下文**动态生成每一轮的用户消息**。因此，对话的生成是逐步、响应式的，更贴近真实用户逐步透露信息、根据助手响应调整问题的行为。

两阶段分工使“要完成什么任务”与“对话如何展开”解耦，既保留任务可验证性（期望工具调用、期望状态与输出），又允许对话形式多样化、自然化。

### 5.3 多Agent协作的蓝图生成

借鉴 LoopTool 的多 Agent 设计，在蓝图生成与交互轨迹生成过程中引入专门角色。

**Planner Agent** 负责设计对话流程。它分析任务意图和可用工具，规划合理的任务分解路径（例如“预订机票”分解为：查询航班、比较价格与时段、选择并确认订单）。Planner 的输出是对话流程图，定义任务完成的必要步骤，而不包含具体对话文案。

**User Agent** 负责生成用户消息。在交互轨迹生成/代理模拟阶段，它根据 Planner 的流程图与当前对话上下文，生成符合用户角色与任务目标的自然语言消息。设计原则包括：不一次性透露完整意图，而是逐步透露相关信息；不编造任务描述之外的信息；根据助手响应动态调整后续问题；当任务目标达成时主动结束对话。

**Assistant Agent** 是实际执行任务的代理，根据 User Agent 的消息决定调用哪些工具、处理返回结果并生成回复。在蓝图生成阶段，Assistant 的响应可作为预定义或参考，用于验证轨迹正确性。

**Tool Agent** 负责模拟工具执行结果。根据工具定义与调用参数，返回符合预期的模拟结果。有真实后端时可调用真实服务；纯模拟环境下返回预设或基于状态的数据。Tool Agent 更适合作为执行与结果组织层，而不应替代数据库状态真值层；在数据库驱动环境中，Tool Agent 只需看到“当前工具调用 + 当前轮局部上下文 + 少量相关摘要”，环境一致性与运行态隔离仍由状态层负责。

Planner、User、Assistant、Tool 四类 Agent 协同工作，使蓝图从“静态脚本”变为“可执行的对话流程定义 + 动态用户行为”。

### 5.4 带验证的迭代式生成

借鉴 APIGen-MT 的多阶段验证机制，蓝图生成过程内嵌验证环节，而非生成完成后统一验证。

**格式验证**：检查任务配置的完整性，如用户意图描述是否清晰、工具调用序列是否可执行、参数是否与工具 Schema 匹配。

**执行验证**：在模拟环境中执行工具调用序列，验证能否产生预期的状态变化。

**对齐验证**：使用 LLM 评估用户意图与工具调用序列的对齐程度，维度包括：正确性（工具调用是否准确反映用户意图）、完整性（用户意图是否被完全覆盖）、满意度（返回结果是否满足用户需求）。

验证失败的任务触发反馈循环：LLM 根据失败原因生成改进建议，重新生成任务配置并再次验证。该过程可迭代多次，直至通过所有验证。据 APIGen-MT 实验，带反馈的生成可将任务配置成功率从约 28% 提升到约 70%。

结合既有设计，状态优先的任务派生与验证检查点生成（验证函数）仍保留：系统根据任务目标推断需检查的数据库状态条件，生成验证函数，供轨迹评估阶段使用。

## 六、代理模拟与轨迹生成

### 6.1 代理框架选型与动态模拟定位

代理模拟阶段选用支持 MCP 协议的工具调用框架，需具备：与 MCP 服务器建立连接并获取工具列表、解析工具输入 Schema 并生成合规调用参数、将工具执行结果融入对话上下文、记录完整对话历史供后续分析。

改进后的代理模拟用**动态用户模拟**替代静态 query 执行：预定义的 query 列表被替换为 LLM 驱动的**用户 Agent**，根据对话上下文与任务目标动态决定每轮的用户消息，从而实现真实的人机交互感。环境仍采用数据库驱动设计，所有状态变更通过 SQLite 记录，轨迹数据除对话历史外仍包含完整状态变更记录，便于程序化验证任务是否真正完成。

### 6.2 动态用户模拟机制

用户 Agent 的 prompt 设计借鉴 APIGen-MT 的核心原则：**不知道**可用工具与 API，仅以自然语言与助手交互；**逐步透露**任务信息，根据当前进展提供必要信息，而非一次性给出完整需求；在**任务目标达成时**生成结束标记，主动结束对话；**不照搬**任务描述原文，而是用自然语言表达相同需求，使对话更流畅。

用户 Agent 配备**自批判机制**：每次生成用户消息后，对消息质量进行评估，维度包括信息完整性（是否包含任务所需关键信息）、表达自然性（是否符合用户角色）、一致性（是否与之前对话矛盾）。仅采纳评分较高的消息；若连续多轮质量不佳，则触发重试或参数调整。

为提高轨迹质量，可采用 **Best-of-N 采样**：每轮需生成用户消息时，并行生成多个候选，通过评分函数（综合消息质量、与任务目标的对齐、与对话历史的连贯性）选取最佳一条作为该轮 user_message。

### 6.3 多轮交互轨迹生成流程

轨迹生成不再按预定义 query 顺序执行，而是**交互式多轮生成**。

**初始化**：加载蓝图中的 system_message 与任务描述，初始化用户 Agent 与助手 Agent 的对话上下文及数据库状态。对于批量并发采集，框架在此阶段还需生成唯一 `run_id`，并将其绑定到当前轨迹的运行上下文。

**每轮交互**：用户 Agent 根据当前对话上下文生成用户消息；助手 Agent 根据用户消息与可用工具决定是否调用工具并生成回复；若需调用工具则通过 MCP 发送请求、获取执行结果并更新对话上下文。该循环持续直到满足以下任一条件：任务完成、达到最大轮次、或用户 Agent 主动结束。

循环中用户消息均为**动态生成**，而非从蓝图中预取，因此能根据助手回复与工具结果做出合理后续反应，形成更自然的多轮对话。

### 6.4 轨迹验证与筛选

生成的轨迹需通过验证方可进入最终数据集，采用**双重标准**。

**基于状态的验证**：比较任务执行前后的数据库状态，检查关键数据是否按预期变更，可与蓝图中的 expected_final_state 及验证函数结合使用。

**基于输出的验证**：比较助手 Agent 的最终回复与蓝图中期望输出，检查是否满足任务需求。

仅当同时通过两种验证的轨迹才被保留。未通过时系统记录失败原因，并视情况处理：工具调用错误可调整参数重试；任务设计问题可反馈至蓝图生成阶段改进任务配置；用户模拟不当可调整用户 Agent 参数重新生成。据 APIGen-MT 数据，轨迹模拟成功率约 67%，结合重试可进一步提高可用轨迹数量。

### 6.5 轨迹数据结构

轨迹数据采用统一 JSON 格式。顶层字段包括：`run_id`、trajectory_id、blueprint_id、skill_name、skill_version、environment_path、created_at，含义与原有方案一致。其中 `run_id` 是运行时隔离键，用于将工具调用日志、状态快照、评估结果与具体轨迹实例关联起来。

主体为 **turns** 数组，每元素对应一轮对话。每轮 **turn** 结构包含：turn_index；**user_message**（本轮用户消息，由用户 Agent **动态生成**）；**user_agent_thinking**（可选，用户 Agent 生成该条消息时的思考过程）；assistant_thinking、assistant_message、tool_calls（工具名称、输入参数、执行结果）、execution_time_ms、state_changes；以及 **interaction_outcome**（可选，该轮交互后的任务完成状态或进度标记）。

为了统一强状态与轻量环境的终态表示，轨迹顶层引入标准化字段 **`final_state`**：

- 对于强状态环境，`final_state` 至少包含 `{"db_snapshot": <run-scoped 数据库快照>}`，与 legacy 的 `final_state_snapshot` 保持兼容；
- 对于轻量环境，`final_state` 直接使用 KV 中维护的 JSON 会话状态（如购物车、预订列表等）；
- blueprint 中的 `expected_final_state` 同样以 JSON 形式给出，两者在评估阶段用于程序化对比和 LLM 辅助判断。

同时保留 **`final_state_snapshot`** 作为兼容字段，用于存放数据库级快照（若存在），但评估和下游消费应以 `final_state` 为主、`final_state_snapshot` 为辅。

## 七、轨迹评估

### 7.1 评估维度设计

轨迹评估采用多维度评分体系，全面衡量合成轨迹的质量。评估维度包括整体质量评分、幻觉风险等级、工具调用合理性、任务完成度、安全性标签和相关性标签。

整体质量评分采用1到5分的标准，5分表示轨迹质量优秀，1分表示质量极差需要重生成。评分考虑因素包括对话是否流畅、工具调用是否恰当、回复是否有帮助等。

幻觉风险等级分为高、中、低三档。高风险表示代理产生了明显的幻觉内容，如虚构不存在的工具、歪曲工具返回结果等。中风险表示代理的回复存在一定的误导性但不至于完全错误。低风险表示代理的回复完全基于工具返回的真实数据。

工具调用合理性评估代理是否正确使用了工具。评估点包括：是否选择了正确的工具、参数是否符合Schema要求、是否正确处理了工具返回的结果、是否忽略或歪曲了工具返回的重要信息。

任务完成度是改进版新增的评估维度，它利用轨迹数据中的final_state_snapshot和验证函数来计算任务完成的精确分数。这种程序化的评估方法比纯LLM判断更加客观和一致。

安全性标签是布尔型的，判断轨迹中是否存在安全风险内容。相关性标签同样为布尔型，判断用户的query与代理的回复是否相关。

### 7.2 评估流程

评估流程采用混合评估方法，结合程序化验证函数和LLM判断。这种设计借鉴了Agent World Model的Code-augmented LLM-as-a-Judge方法，同时利用了轨迹数据中包含的数据库状态快照。

程序化验证阶段首先执行。评估器加载轨迹数据中的验证函数（如果蓝图生成阶段生成了验证函数），将final_state_snapshot作为输入，计算任务完成分数。对于共享数据库 + `run_id` 的环境，程序化验证还应首先检查 `run_id` 与快照、日志是否一致，以确保评估针对的是单条轨迹的运行态，而非全局混合状态。程序化验证的优势在于结果完全确定，不受LLM随机性的影响。

LLM评估阶段在程序化验证的基础上进行补充评估。对于程序化验证难以衡量的维度（如对话流畅性、回复有用性），使用LLM进行评估。LLM评估的输入包括完整的轨迹数据和程序化验证的结果，输出的结构化评分包含整体质量评分、幻觉风险等级、工具调用合理性、安全性标签和相关性标签。

### 7.3 评估结果应用

评估结果作为数据合成的质量门，用于过滤低质量轨迹。预设质量阈值为3分，低于该阈值的轨迹需要进行重生成或人工审核。高风险幻觉或安全问题的轨迹无论整体评分如何都需要标记为不可用。

任务完成度评估结果有特殊的应用场景。由于程序化验证可以精确计算任务完成分数，系统可以根据分数将轨迹分类为完全完成、部分完成和未完成。部分完成的轨迹虽然整体质量可能不错，但任务目标的达成程度有限，可能需要进一步优化。

评估结果同时用于生成改进建议。分析评估反馈中的缺点和改进点，将这些信息汇总后可以用于指导蓝图的优化、工具Schema的改进、或者环境生成的调整，形成数据合成的闭环优化。

## 八、有状态环境的处理策略

### 8.1 状态管理架构

数据合成场景中的有状态环境涉及多种状态类型，包括文件系统状态、网络连接状态和数据库状态。改进后的方案采用统一的数据库作为状态后端，将各类状态都抽象为数据库中的数据记录，从而实现一致的状态管理。

这种设计解决了有状态环境面临的几个核心挑战。首先是状态污染问题：可以通过为每个任务分配独立的数据库实例实现物理隔离，也可以在共享数据库内通过 `run_id` 实现逻辑隔离。后者特别适合只读或轻状态、高并发的数据合成场景：共享静态表保存所有任务共用的只读种子数据，运行态表按 `run_id` 分区保存每条轨迹的会话元数据、工具调用日志与快照。其次是不可复现问题：通过使用确定性随机算法生成初始数据，相同的任务ID总是产生相同的初始数据或共享静态状态。第三是状态验证问题：通过数据库事务日志和 run-scoped 快照，可以精确追踪和验证状态变更。

### 8.2 文件系统操作模拟

针对文件系统操作的需求，改进后的方案将文件系统抽象为数据库中的记录进行管理，而非直接操作真实文件系统。

具体实现方式是在数据库中定义虚拟文件系统Schema，包含以下表结构：files表记录文件元数据（路径、名称、大小、创建时间、修改时间等）；file_content表记录文件内容（二进制或文本）；directories表记录目录结构。这种抽象使得文件操作可以像数据库操作一样进行事务管理和状态追踪。

工具执行时的文件操作被转换为数据库操作。例如，读取文件操作被转换为查询file_content表；写入文件操作被转换为更新或插入file_content表记录。这种设计的优势在于：文件状态变更可以被精确追踪、可以实现版本控制（通过保留历史记录）、便于验证文件操作的正确性。

对于需要访问真实文件系统的场景（如代码仓库、配置文件等），可以采用混合模式：真实文件系统的特定目录被映射到数据库中的虚拟文件系统，工具操作针对虚拟文件系统进行。

### 8.3 外部服务的状态初始化

针对涉及外部服务依赖的场景（如社交媒体数据、API调用等），采用基于唯一标识符的数据库初始化方案。这一设计在原有方案的基础上做了增强，与数据库驱动的环境架构完全融合。

技术实现上使用SQLite作为统一的数据库后端。对于强状态任务，每个任务可以对应一个独立的数据库文件；对于批量并发的轻状态或只读任务，也可以使用一份共享数据库，将静态参考数据集中存储，并用 `run_id` 将每次合成的运行态数据分区隔离。数据模板定义了场景涉及的数据实体（如用户、推文、订单等）和生成规则。生成规则使用确定性随机算法，保证相同任务ID产生相同的初始数据，确保可复现性。

数据模板的设计更加系统化。完整的数据模板包含以下内容：实体定义描述每个数据实体的类型、属性和约束；关系定义描述不同实体之间的关联方式；初始数据配置描述初始状态下各实体应该包含的数据；任务派生规则描述如何从初始数据中派生可行的任务。

### 8.4 执行流程整合

有状态环境的处理整合到代理模拟阶段的任务初始化中。具体流程如下：任务开始前，根据任务唯一ID初始化专属的数据库实例，或在共享数据库模式下创建新的 `run_id` 运行上下文并关联运行态表；代理执行过程中，所有涉及状态变更的操作都通过状态访问层进行，状态变更被记录到数据库中；任务执行完成后，可以选择保留数据库状态或 run-scoped 快照用于调试分析，或者直接删除数据库实例 / 清理该 `run_id` 的运行态记录。

这种设计的优势在于：完全隔离了不同任务之间的状态，确保每个任务的执行环境都是干净的和可复现的；数据库提供了强大的查询和验证能力，便于实现精确的轨迹评估；状态变更的完整记录使得问题排查和审计变得简单。

## 九、流水线工程实践

### 9.1 阶段间数据格式

各阶段之间的数据交互采用JSON格式作为标准。这种选择基于几方面考虑：JSON是广泛使用的数据格式，便于各语言和框架的解析处理；JSON具有良好的可读性，便于调试和人工审查；JSON的结构可以灵活扩展，适应不同场景的需求。

具体的数据格式定义使用JSON Schema进行形式化。Blueprint的结构、轨迹数据的Schema、评估结果的格式等都需要预先定义JSON Schema。JSON Schema不仅用于验证数据结构的正确性，还可以用于自动生成文档和类型定义。

改进版流水线新增与调整了以下数据格式定义。**环境元数据格式**包含环境名称、类型、数据库 Schema 路径、工具列表路径、初始数据路径等字段。**状态空间规划格式**包含实体类型列表、属性定义、关系定义和操作映射等字段。**验证函数格式**包含验证逻辑代码、输入参数定义和输出格式等字段。

**蓝图格式**（对应第五章两阶段与多 Agent 设计）分为任务配置与交互生成配置：任务配置含 task_id、goals、possible_tool_calls（可能的工具调用）、expected_final_state；交互生成配置含 user_agent_config（角色、性格、知识边界等）、end_condition，以及可选的 initial_state。**轨迹格式**（对应第六章动态用户模拟）中，每轮 turn 的 user_message 为动态生成内容，并增加 user_agent_thinking、interaction_outcome 等可选字段，其余字段与 6.5 节描述一致。对支持并发采集的环境，轨迹顶层还应增加 `run_id`，并约定 `final_state_snapshot` 与评估结果都围绕该 `run_id` 组织。

### 9.2 错误处理与重试机制

流水线执行过程中可能出现各类错误，需要建立完善的错误处理和重试机制。对于可恢复的错误（如网络超时、临时服务不可用），设置重试策略，在一定次数后仍然失败则标记为失败并记录原因。对于不可恢复的错误（如环境验证失败、生成代码语法错误），直接标记失败并输出详细错误信息。

每个阶段的执行都应该记录完整的日志，包括输入参数、执行过程、输出结果和执行耗时。日志的详细程度应该足够支持问题的排查，但也要避免过度日志导致的存储开销。日志应该与任务标识关联，便于通过标识检索完整的执行历史。

改进版的错误处理增加了专门针对数据库错误的处理逻辑。常见的数据库错误包括：Schema与工具操作不匹配导致的数据操作失败、初始数据不完整导致的空引用错误、事务冲突导致的并发问题等。针对这些错误，系统会生成详细的诊断信息，帮助定位问题根源。

### 9.3 并行执行与资源管理

大规模数据合成任务需要支持并行执行以提升吞吐量。并行化可以在多个维度进行：多个Skill的环境生成可以并行进行；同一Skill的多个Blueprint生成可以并行进行；多个轨迹的代理模拟和评估可以并行进行。

并行执行需要考虑资源隔离和限制。每个并行任务应该运行在独立的容器或进程中，避免状态污染。CPU、内存和磁盘的使用需要设置上限，防止单个任务耗尽资源导致系统不稳定。对于使用外部API的任务，需要控制并发速率避免触发API限制。

改进版的并行执行需要特别关注数据库隔离。对于复杂写操作环境，可以继续使用“每任务独立数据库实例”的物理隔离模式；而对于只读或轻状态、高并发的合成环境，推荐使用“单容器 + 共享数据库 + `run_id` 逻辑隔离”的模式：共享静态表保存只读参考数据，运行态表按 `run_id` 记录工具调用日志、轨迹输出和状态快照。无论采用哪种模式，数据库实例或 run-scoped 记录的生命周期管理都需要纳入资源管理框架，确保任务完成后可以准确保留或清理对应状态。

### 9.4 蓝图与代理模拟的工程实现考量

多 Agent 协作与动态用户模拟在工程上需额外考虑以下方面。

**Agent 间状态管理**：Planner、User、Assistant、Tool 四类 Agent 需清晰的状态边界与数据流，避免上下文混淆或状态污染。例如用户 Agent 仅接收对话历史与任务意图，不暴露工具 Schema；Tool Agent 的输入输出与 MCP 调用约定一致，便于替换真实环境。

**生成效率与成本**：多轮交互与 Best-of-N 采样会增加调用次数与延迟。可在质量与成本间做权衡：例如限制 max_turns、对部分轮次使用单次采样、或对用户消息质量设置可接受的阈值以减少重试。

**错误处理与鲁棒性**：动态生成中可能出现超出预期的用户表达、助手决策异常或工具失败。需定义超时、最大重试次数、异常分类与降级策略（如记录失败原因并反馈至蓝图或跳过该轨迹），并保证单条轨迹失败不影响同批次其他任务。

### 9.5 可观测性与监控

流水线的运行状态需要具备良好的可观测性。关键指标包括：各阶段的执行成功率、执行耗时分布、队列长度和积压情况、资源使用率等。

改进版的可观测性还需要关注环境质量的监控。监控指标包括：环境生成成功率、各层验证的通过率、双Agent评估的平均分数、任务可解率等。这些质量指标可以帮助识别流水线中的系统性问题，为优化方向提供数据支持。

建议部署监控面板展示实时指标，设置告警机制在异常发生时及时通知。执行日志应该与任务标识关联，便于通过标识检索完整的执行历史。对于批量数据合成任务，建议生成每日或每周的质量报告，总结生成数据的数量、质量分布和主要问题。

## 十、总结与展望

本技术路线方案详细阐述了改进后的数据合成流水线设计，涵盖从环境生成到质量评估的全流程。方案的核心特点包括：基于四阶段渐进式合成的环境生成架构、SQLite 数据库驱动的状态管理、POMDP 形式化环境建模、自验证与自动修复机制、双 Agent 质量评估体系、状态优先的任务生成策略、以及程序化验证与 LLM 判断相结合的轨迹评估方法。在蓝图与代理模拟方面，采用两阶段蓝图生成（任务配置 + 交互轨迹生成）、多 Agent 协作（Planner / User / Assistant / Tool）、带验证的迭代式生成，以及代理模拟阶段的动态用户模拟、多轮交互与轨迹双重验证，使合成数据更贴近真实人机对话并保持可验证性。

这些改进使得流水线在以下方面有了显著提升。在规模化能力方面，四阶段架构和自动化验证机制使得大规模环境生成成为可能，根据Agent World Model的实验数据，单一pipeline可以生成数千个高质量环境。在生成质量方面，多层次的质量门控和自修复机制确保了生成代码的可靠性，首次成功率超过85%。在可验证性方面，数据库驱动的状态管理使得轨迹评估可以采用程序化验证函数，避免了纯LLM判断的不一致性问题。在可复现性方面，确定性随机算法和任务隔离机制确保了每次执行的结果都可复现。

后续可以探索的优化方向包括：引入更智能的提示词自动优化机制，根据生成效果反馈迭代提示词设计；建立更丰富的示例库，覆盖更多Skill类型和场景；探索基于强化学习的轨迹质量优化，根据评估反馈自动调整生成策略；构建人工审核与机器评估的混合质量控制流程；以及研究跨领域Skill的知识迁移方法，利用已生成环境的信息辅助新环境的生成。
